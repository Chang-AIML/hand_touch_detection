# Mixed multi-dataset spotting — architecture, data & reproduction (2×RTX Pro 6000)

Everything you need to sync this to another box (2× RTX Pro 6000, 96 GB each, no
Slurm/k8s — just `torchrun`) and run the same balanced mixed-dataset training.
All code lives under `hand_touch_spotting/dpc/` (self-contained; the rest of the
repo is the original HOI4D project it builds on).

---

## 0. TL;DR

```
frames (tar OR loose jpg)
  → WindowedSpottingDataset       # cut clips into ≤W-frame windows, domain-balanced
  → OnlineVJEPA.extract_grid()     # frozen V-JEPA 2.1 ViT-B, ON-THE-FLY (no feature cache on disk)
      → (ceil(T/2), 576, 768) fp16
  → FrameCompress (trainable)      # 8 query tokens → N motion tokens in Qwen embedding space
  → frozen Qwen3-VL-8B (idx-decode)# LLM GENERATES the event frame index as digit tokens
  → CE loss on the answer digits
```
Trainable = **FrameCompress only (27.5 M)** in Stage 1. Frozen = V-JEPA + Qwen.
Stage 2 adds **LoRA** on Qwen. Eval = per-window spotting **mAP@{0,1,2,4}**, reported
**per dataset**, on **val** (test held out).

**On 2×Pro 6000 the memory story is easy** (96 GB ≫ 40 GB A100): use `--window_frames 600`,
`--batch_size 2-4`, no `expandable_segments`, no FSDP. Just `torchrun --nproc_per_node=2`.

## 1. Model architecture

Winning "idx-compress" stack (unchanged from the base project, we only feed it new data):

- **V-JEPA 2.1 ViT-B (384px, frozen)** — per-frame visual encoder. On-the-fly, no grad,
  runs **fp32** (required to bit-match the original offline cache; the RoPE path needs fp32).
  Output = "even-pass grid" `(ceil(T/2), 576, 768)` (576 = 24×24 spatial tokens, 768-dim).
- **FrameCompress (trainable, 27.5 M)** — 8 learnable query tokens cross-attend the 576
  patch tokens per frame → 1 "motion token"/frame, projected to Qwen's 4096-dim and
  RMS-matched to its embeddings. Interpolates ceil(T/2)→T internally. Config we use:
  `n_q=8, stain=0, use_text=0, gate_lang=0` (pure-visual — the winning "comp_notext").
- **Qwen3-VL-8B-Instruct (frozen)** — the LLM. We feed `inputs_embeds` directly:
  `[instr][ f₀ digits ][M₀][ f₁ digits ][M₁]…[ question + " Answer:"][ answer digits + EOS ]`.
  `use_idx=True` puts each frame's index as **digit tokens before its motion token** (the
  "copy handle"). `use_anchor=True` (**keep** the frozen Qwen-ViT anchors — the base
  `comp_notext` was trained WITH them; dropping them cost HOI4D ~40 mAP points). Anchors are
  sampled **window-locally from the decoded window frames** (`idx_localizer._anchors_for_secs`
  with `frames=`), so there is **no loose-jpg glob and no on-disk frame dir needed** — the
  tar/DirFrameSource window already carries the pixels. `--anchor_stride 5 --anchor_max_side 252`.
  Training = teacher-forced CE on the answer digits+EOS. Inference = greedy-generate the
  index, regex-parse the int → predicted frame(s), score = mean softmax of its digit tokens.

**Two-stage (LLaVA-style):**
| | trainable | LLM | init |
|---|---|---|---|
| **Stage 1** | FrameCompress | frozen | `comp_notext_best.pt` (HOI4D Stage-1 connector) |
| **Stage 2** | FrameCompress + **LoRA** | unfrozen (LoRA r16) | Stage-1 mixed connector |

## 2. Data organization

### 2a. Task & splits
Multi-class **point-event spotting**: given a natural-language question about an event
type, predict the frame index(es) where it occurs. Metric: spotting mAP@δ (δ frame tolerance).

- **In-domain (trained + in-domain val):** `touchmoment_hoi4d`, `touchmoment_taco`,
  `tennis`, `finegym`, `soccernet_ball`, `fs_perf`.
- **Out-of-domain (NEVER trained; zero-shot val):** `finediving`.
- Datasets with NO extracted frames are excluded: `fs_comp`, native-TouchMoment.
- Train on `train` split, monitor on `val`, **hold `test` for the final report.**

### 2b. Annotation format (converted)
`dpc/make_dpc_annotations.py` converts the raw per-clip split json
`{video, num_frames, events:[{frame,label,comment}], fps, width, height}` into the training schema:
```json
{"video_id": "<dataset>/<clip>", "dataset": "<registry_key>", "fps": .., "num_frames": ..,
 "events": [{"type": "<label>", "frame": <int>, "question": "<NL question>"}]}
```
- `video_id` = `<video_prefix>/<clip>` and MUST match the FrameSource key (and, on DPC, the
  `index.sqlite` clip id). TouchMoment/HOI4D & TACO both use prefix `TouchMoment`.
- Natural-language `question` per event is generated by `dpc/questions_multi.py`
  (`question_for(dataset,label,phase,ordinal,count)`): curated bank for HOI4D touch/untouch,
  templated (humanized-label) phrasings for the rest, with a **train/test phrasing split**
  (eval uses HELD-OUT phrasings).
- Registry (which datasets, in/out-domain, TouchMoment sub-parts) is `REGISTRY` in
  `make_dpc_annotations.py` — edit there if paths differ on the new box.

Run it once:
```bash
python -m dpc.make_dpc_annotations --root <ACTION_SPOTTING_ROOT> --out <ANN_DIR>
```

### 2c. Frames — FrameSource (two backends, same API)
`dpc/frame_source.py`:
- **`DirFrameSource(root)`** — loose jpgs at `<root>/<dataset>/<framesdir>/<clip>/*.jpg`
  (`framesdir='Frames'` for TouchMoment else `frames`). **Use this on the Pro 6000 box**
  if the dataset is stored as loose frames (pass `--local_frames <root>`).
- **`TarFrameSource(root, index_sqlite)`** — reads the DPC tar shards via `as_dataset.TarStore`
  + `index.sqlite`. (This is the DPC layout; only needed if you copy the packed dataset.)
Both expose `load_window(video_id, start, end) -> uint8 (T,H,W,3)`.

### 2d. Windowing + domain-balanced sampling  (`dpc/windowed_dataset.py`)
- **Fixed non-overlapping grid**, window ≤ `W` frames (`stride=W`). NOT centered on GT —
  events fall at arbitrary positions. Clip < W → one window = whole clip. Long clips
  (finegym/soccernet) → many windows. **`W` is the hard per-clip frame cap** that keeps the
  Qwen sequence bounded (600 frames ≈ 2350 tokens; well within Qwen's context).
- One sample per **(window × event-type-present)**; a `negative_rate` fraction of
  event-type-absent windows are emitted as `'none'` targets (so the model learns "no event").
- **`balance_per_dataset=N`**: caps each dataset to N (window,type) samples, spread evenly
  across its clips (round-robin) → fixes domain imbalance (finegym has ~22 k windows vs
  soccernet's few) and starvation of few-clip datasets. We use **N=2500** (≈ balanced 11.6 k
  train windows, ~4.8 M frames; without it finegym is 65 % of the data).

### 2e. On-the-fly V-JEPA  (`dpc/vjepa_online.py`)
`OnlineVJEPA(device, dtype=torch.float16, repo=$VJEPA_REPO, ckpt=$VJEPA_CKPT)` — loads the
frozen encoder once; `.extract_grid(frames_uint8) -> (ceil(T/2),576,768)`. Bit-exact to the
original offline `grid_even` cache (verified). We compute it **in the training loop**
(frozen/no_grad) instead of caching to disk — caching all grids would be multi-TB.

### 2f. Eval scorer  (`dpc/eval_vendor/`)
Vendored from the sibling `hand_touch_detection` repo (`compute_mAPs`, `maps_quiet`) so it's
self-contained. `maps_quiet(truth, pred)` → `[mAP@0,1,2,4]` (%), where truth/pred are
`list[{video, events:[{label,frame(,score)}]}]`.

## 5. Run it on 2×RTX Pro 6000

```bash
# --- env (edit paths to the new box) ---
export QWEN_PATH=/path/to/Qwen3-VL-8B-Instruct
export VJEPA_REPO=/path/to/repos/vjepa2
export VJEPA_CKPT=/path/to/vjepa2_1_vitb_dist_vitG_384.pt
export AS_DATA_ROOT=/path/to/Action_Spotting            # loose-frame dataset root
export AS_ANN_DIR=/path/to/annotations_dpc              # output of make_dpc_annotations
export AS_OUT=/path/to/runs
export HF_HOME=/path/to/hf_cache
export WANDB_API_KEY=<your key>                         # or WANDB_MODE=offline
cd hand_touch_spotting

# --- one-time: build converted annotations ---
python -m dpc.make_dpc_annotations --root "$AS_DATA_ROOT" --out "$AS_ANN_DIR"

# --- STAGE 1 (frozen LLM, connector only) : 2 GPUs, big window/batch (96GB each) ---
torchrun --standalone --nproc_per_node=2 dpc/train_mixed.py \
  --run_name mixed_stage1 --wandb_mode online --wandb_project dpc_mixed_spotting \
  --local_frames "$AS_DATA_ROOT" \
  --init_fc outputs/idx_compress/comp_notext/best.pt \
  --stain 0 --use_text 0 --n_q 8 --gate_lang 0 \
  --use_anchor 1 --anchor_stride 5 --anchor_max_side 252 \
  --window_frames 600 --jitter 120 --vjepa_bf16 1 \
  --batch_size 2 --grad_accum 4 \
  --lr 2e-4 --warmup 20 \
  --balance 2500 --eval_balance 60 --epochs 3 --max_steps 800 \
  --eval_every 100 --eval_windows 400 --negative_rate 0.15

# --- STAGE 2 (unfreeze via LoRA) : from Stage-1 connector ---
torchrun --standalone --nproc_per_node=2 dpc/train_mixed.py \
  --run_name mixed_stage2 --wandb_mode online --wandb_project dpc_mixed_spotting \
  --local_frames "$AS_DATA_ROOT" \
  --init_fc "$AS_OUT/mixed/mixed_stage1_best.pt" \
  --lora 1 --lora_rank 16 --stain 0 --use_text 0 --n_q 8 --gate_lang 0 \
  --use_anchor 1 --anchor_stride 5 --anchor_max_side 252 \
  --window_frames 600 --jitter 120 --vjepa_bf16 1 \
  --batch_size 2 --grad_accum 4 \
  --lr 2e-4 --lr_lora 1e-4 --warmup 20 \
  --balance 2500 --eval_balance 60 --epochs 3 --max_steps 800 \
  --eval_every 100 --eval_windows 400 --negative_rate 0.15
```

Notes for the Pro 6000 box (vs the 4×A100 DPC run):
- `--nproc_per_node=2` (2 GPUs). Effective batch = `batch_size × 2 × grad_accum`.
- **96 GB ⇒ raise `--batch_size` (2–4) and `--window_frames 600`**; drop `grad_accum` to keep
  the effective batch similar. No `PYTORCH_CUDA_ALLOC_CONF` / FSDP needed.
- `--use_anchor 1` keeps the frozen Qwen-ViT anchors (base was trained with them). Anchors are
  sampled from the decoded window frames — **no loose-jpg dir required** for anchors.
- `--jitter 120` = train-only window-start jitter (breaks the "event always mid-window" prior;
  auto-skipped on clips shorter than the window, i.e. pre-cut datasets). `--vjepa_bf16 1` runs
  the V-JEPA encoder under autocast(bf16) — ~2× faster, RoPE stays fp32.
- `--local_frames` switches to `DirFrameSource` (loose jpgs). Omit it to use the tar+`index.sqlite`.
- `eff_batch`, per-step `loss` + `grad_norm`, and per-dataset `val/<split>/<dataset>_mAP@{0,1,2,4}`
  are logged to wandb every step / every eval. A `<run>_metrics.csv` is written under `$AS_OUT/mixed`.

### Key knobs
`--balance` (windows/dataset) · `--window_frames` (≤ frame cap) · `--negative_rate` ·
`--lr` / `--warmup` · `--lora` (Stage 2) · `--eval_every` / `--eval_balance` · `--max_steps` ·
`--jitter` (train-only aug) · `--vjepa_bf16` · `--fsdp` (40GB-A100 only; skip on Pro6000).

> Platform note (DPC A100 node): `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True` prints
> "not supported on this platform" and is ignored there — so win600+anchor OOM was true capacity,
> hence `--fsdp`. Harmless on Pro6000 (and unneeded).

## 6. Gotchas we already hit & fixed (don't re-discover them)

1. **Eval mode depends on FSDP.** Without FSDP: shard eval across ranks + `all_gather` (rank-0-only
   eval + barrier would idle ranks 1-3 → NCCL watchdog crash). **With `--fsdp`**: sharded eval
   DEADLOCKS — sharded weights make every generate-forward a collective all-gather, and uneven
   per-rank shards + early-EOS desync the collectives (fast rank hits `all_gather_object` while a
   slow rank is still gathering weights → 30-min NCCL timeout → SIGABRT). Fix in `evaluate()`:
   under FSDP run **replicated lockstep eval** — every rank generates the SAME windows (identical
   greedy forwards → identical all-gather order → lockstep) and only rank0 records. NCCL timeout is 30 min.
2. **fps varies (15–30), and windowing is by FRAME.** So a fixed frame budget = different SECONDS
   per dataset (600f = 40s@15fps but 20s@30fps). We keep native per-clip fps (no resampling — user
   decision); the model's `fps` groups seconds for anchor placement (`int(round(clip_fps))`, never 15).
   Short pre-cut sets (hoi4d 300f / taco 153f / finediving 101f / tennis 333f) fit whole in one window;
   only finegym / fs_perf / soccernet are long enough for the window to bound temporal coverage.
2b. **transformers ≥4.57 Qwen3-VL `get_image_features` returns a 2-tuple**
   `(image_embeds_per_image, deepstack_image_embeds)`, not a single tensor. `wrapper.vit_anchor_groups`
   takes `out[0]`. If you pin a different transformers, re-check this return shape or anchors crash
   with `'tuple' object has no attribute 'to'`.
3. **Loss is noisy per-step (1.2–2.3)** because each step mixes 6 heterogeneous datasets;
   watch the running avg + the **per-dataset mAP**, not single-step loss. `lr=5e-4` made loss
   *rise* (base disruption) — `lr=2e-4 + warmup` fixed it (grad-norm ~0.3–0.8, stable).
4. **Stage-1 loss plateaus high** (~1.3–1.4) — frozen-LLM capacity limit; the real gains
   need Stage 2. Judge by mAP, not loss.
5. **soccernet_ball has only 4 train / 1 val / 2 test clips** → its per-dataset val is basically
   noise; weight it lightly. finediving (OOD) val=450/test=749 is solid.
6. On A100 (40 GB) window=600 **training backward** OOMs even at bs=1 (eval fits): the frozen
   16GB bf16 Qwen replicated per-GPU leaves too little for the win600+anchor backward graph.
   Fix on 40GB = `--fsdp 1`: FSDP FULL_SHARD **only the `Qwen3VLTextDecoderLayer`s** (frees
   ~12GB/GPU), keeping `embed_tokens` (read by the connector) and the ViT (anchor path) FULL via
   `ignored_modules` / by wrapping only the text submodule. Frozen base still all-gathers in
   backward (output requires grad; torch≥2.1 PR#101982) so input-grads reach the connector; the
   connector stays replicated + manually all-reduced. Grad-ckpt must be `use_reentrant=False`.
   **On Pro 6000 (96 GB) skip `--fsdp` entirely** — the 16GB base fits with huge headroom; just
   raise `--batch_size`/`--window_frames`. FSDP would only add all-gather overhead there.
7. **Anchors were briefly dropped (`use_anchor=0`) to dodge the loose-jpg glob — DON'T.** It cost
   HOI4D ~40 mAP (67→26). The fix: sample anchors window-locally from the decoded frames
   (`_anchors_for_secs(frames=…)`), so anchors work from tars with no on-disk frame dir. `use_anchor=1`.
